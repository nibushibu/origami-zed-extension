# Syncing with upstream

This extension is a fork/port of the official VS Code extension for Origami:
https://github.com/WebOrigami/origami-vscode-extension

It's tracked as a git submodule at `upstream/`, pinned to a specific commit.
To pull in upstream fixes/changes:

```sh
git submodule update --remote upstream   # move upstream to its latest commit
node scripts/sync-upstream.mjs           # regenerate derived files (see below)
git diff                                 # review what changed
git add -A && git commit                 # commit once you're happy with the diff
```

`scripts/sync-upstream.mjs` also writes `scripts/upstream-lock.json`, which
records the upstream commit/version this repo was last synced against. This
is what lets the sync script warn you when upstream's TextMate grammar
changes (see below).

## Generated vs. hand-written files

To keep upstream syncing mechanical and conflict-free, every file in this
repo falls into exactly one of two categories. **Never hand-edit a generated
file** — your edits will be silently overwritten the next time the sync
script runs.

### Generated (regenerated by `scripts/sync-upstream.mjs`, do not hand-edit)

| File(s) | Derived from |
| --- | --- |
| `language-server/vendor/**` | `upstream/src/server/**`, `upstream/src/utilities.mjs`, `upstream/src/client/builtins.json` — copied verbatim. This code is editor-agnostic LSP logic, so no translation is needed. |
| `language-server/package.json` | `upstream/package.json`'s `dependencies` (filtered to the packages the vendored server code actually imports) |
| `languages/origami/config.toml` | `upstream/package.json`'s `contributes.languages` entry for `origami` + `upstream/language-configuration.json` |
| `languages/origami-html/config.toml` | same, for `origami-html` |
| `languages/origami-markdown/config.toml` | same, for `origami-markdown` |
| `scripts/upstream-lock.json` | bookkeeping (commit SHA, upstream version, TextMate grammar hash, timestamp) |

### Hand-written (Zed-specific, the sync script never touches these)

| File(s) | Notes |
| --- | --- |
| `extension.toml`, `Cargo.toml`, `src/lib.rs` | The Rust/WASM wrapper. Launches `language-server/zed-entry.mjs` as an external process; see doc comments in `src/lib.rs`. |
| `language-server/zed-entry.mjs` | Composes the vendored modules into a single LSP server process, and reproduces upstream's client-side builtins-completion behavior (which VS Code's client registers separately, but Zed extensions can't). See doc comment at the top of the file for why this exists. |
| `grammars/origami/**` | Placeholder Tree-sitter grammar (see `grammars/origami/README.md`). VS Code's TextMate grammar (`upstream/syntaxes/*.tmLanguage.json`) has no automatic equivalent in Tree-sitter, so this must be maintained by hand. |
| `languages/*/highlights.scm`, `indents.scm`, etc. (once added) | Tree-sitter queries; same reasoning as above. |

## What to check after every sync

1. Run the sync script and review `git diff` as usual — most upstream
   changes (bug fixes in autocomplete/definition/diagnostics logic, new
   builtins, dependency bumps) should show up as clean, mechanical diffs
   under `language-server/vendor/`, `language-server/package.json`, and
   `languages/*/config.toml`.
2. If the sync script prints a warning about `syntaxes/*.tmLanguage.json`
   having changed upstream, manually review what changed there and consider
   whether `grammars/origami/grammar.js` and any `languages/*/*.scm` query
   files need a matching update.
3. If `language-server/zed-entry.mjs` fails to import something from
   `language-server/vendor/**` (e.g. a function was renamed or its signature
   changed upstream), you'll need to hand-update `zed-entry.mjs` to match.
   This is the one seam where an upstream change could require a manual
   code fix rather than just a data/config regeneration.
4. Bump `version` in both `extension.toml` and `Cargo.toml` (they must match)
   when you cut a new release.
